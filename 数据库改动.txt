1. 总体目标与边界（V1.5）
1.1 目标

首页变为一体化工作台：分级汇总表 +（可折叠）趋势图（原技术仪表盘功能并入）

情报全流程升级为**“先填全字段 → 提交 → AI 推荐指标/预警判定 → 人工修正 → 预览 → 发布”**

支持人为新增/编辑预警（发布页与情报总表）

情报总表：更强筛选（领域/来源/类型/重要度/新旧技术/comment/预警）+ 关键词高亮与正文片段展开

上传：支持多领域归属、草稿管理、Excel 批量上传（含模板）

1.2 边界（仍可验收）

统计/趋势/预警口径只基于 published 情报（草稿不入统计）

文献库仍作为只读来源（可转情报草稿，不改变主闭环）

AI：仅做“推荐指标 + 预警建议 + 可解释说明”，最终入库由人工确认

2. 信息架构（页面级）

你现在的页面会调整为 6 个（其中“技术仪表盘”并入首页，不再单独导航）：

index.html 首页工作台

左：时间筛选（周/月/年 + from/to）+ 领域树/快速跳转

中：分级汇总表（下钻/展开 top10）

右（可折叠面板）：趋势图（自选 X/Y 轴）

支持“一键进入情报总表（带筛选）”

info_table.html 情报总表（主工作界面）

左侧筛选器（可隐藏）

顶部：领域快速跳转（输入可选）

搜索框在下一行（你要求）

列表：关键字高亮 + 正文片段（前后展开）+ comment/预警状态标识

支持：人为新增/编辑预警入口

upload_review.html 上传&复核（主闭环）

第一步：填写全部字段（正文在 tags 上面；支持多领域选择）+ comment（上传前）

点击提交：同页出现 AI 推荐指标/预警建议

人工调整指标值/预警状态

若预警 active：强制填写预警内容/结论

预览 → 发布

drafts.html 本地草稿管理（参考总表 UI）

列表/筛选/编辑/删除/继续提交 AI

草稿存在 LocalStorage（样机）或 DB（上线）

excel_import.html Excel 批量上传

下载模板按钮

上传 Excel → 预览解析结果 → 校验错误提示 → 批量生成草稿 → 进入 drafts 管理或逐条提交 AI

info_detail.html 情报详情（已有）

增加：comment 展示、预警编辑入口（权限控制）

3. 关键交互与业务流程（按你最新口径）
3.1 首页左侧时间筛选（去掉按钮）

UI：

快捷：近 7 天 / 近 30 天 / 近 365 天 / 自定义

自定义：from/to 仍可编辑

行为：任何变动都自动刷新（不再需要“应用筛选按钮”）

实现细节：

点击快捷选项会同时写入 from/to

from/to 手改会自动触发 render（加 300ms debounce 防抖）

3.2 人为增加预警（发布页 & 情报总表）

你要的是：即便不是新技术/超范围，也能人工开预警。

规则建议：

预警入口（按钮）放在：

上传&复核：AI 结果出来后，允许“强制开启预警”

情报总表：每条情报行上有「预警」按钮（无预警→创建；有预警→编辑/解除/延期）

“是否预警”从自动判定 + 人工覆盖组合：

alert_source = auto|manual

manual_override = true/false

若人工关闭预警，需要填写 manual_dismiss_reason

3.3 上传时可选择多个领域

你现在从单 domain_id 变成 多选 domain_ids：

业务含义：

一条情报可同时属于多个 4 级领域（或允许跨层级但仍建议落到最深层）

统计口径：在各领域树下都计数（注意“总计”会双计，所以下面给你一个避免坑的方案）

统计避免坑（强烈建议）

首页“分级汇总表”的口径要明确：

领域统计 = 去重后的情报数（以情报 id 去重）

或者 领域统计 = 关联数（允许重复计数）

老板一般更喜欢“去重后的情报数”，否则总数对不上。

我建议：

领域行显示：unique_info_count（去重）

鼠标 hover 显示：relations_count（关联次数）

3.4 上传页字段顺序与 AI 阶段

你要的闭环是 “先填完 → 提交 → AI → 人工改 → 预览发布”，并增加“上传前 comment”。

新流程（强制）

填写所有字段（含 comment、领域多选、tags、多选来源/类型、重要度、新旧技术）

点击【提交并生成 AI 建议】

页面下半区出现：

AI 推荐指标（候选列表 + 建议值/单位/来源片段）

AI 预警判定：是否预警、原因（新技术/超范围/人工）

人工修正：

勾选/删除指标、修改数值与单位

选择是否预警

若预警=是：强制填写（预警标题、内容、reviewer_comment、duration）

预览（只读）

发布（写入 published，参与统计）

3.5 情报总表增强（筛选、搜索、高亮、展开）

你提出的点我逐条落地成 UI/行为：

左侧筛选（可隐藏）

领域筛选（树形 + 快速跳转输入可选）

新技术/已有技术（is_new_tech）

来源（source_types 多选）

情报类型（info_types 多选）

重要度（importance_score 范围或多选）

是否预警（has_active_alert）

是否有 comment（has_comment）

时间范围（同首页）

搜索框放下面一行

上一行放“领域快速跳转”

下一行单独放 keyword 搜索（标题+正文+comment）

关键词高亮 + 正文前后展开

展示规则（列表每条情报）：

标题中命中词高亮

正文显示命中附近片段：…前 60 字 + <mark>命中</mark> + 后 60 字…

点击「展开上下文」→ 展开为 400~800 字（仍是片段，不是全文）

点击「全文」→ 进详情页

实现：前端对 content 做一次 indexOf 定位，切片并 wrap <mark>；多命中显示第一个为主（或允许多段）。

3.6 技术仪表盘并入首页（只保留画图趋势、可折叠）

首页右侧/下方加一个可折叠面板：

“趋势图面板”能力

可选 X 轴：时间（published_at）/ 重要度 / 其他可扩展字段

可选 Y 轴：某个指标 value_number（或 count）

支持：

折线（时间 vs 指标值）

柱状（领域 vs 情报数）

散点（时间 vs 指标值，点颜色=预警颜色）

自动折叠：默认收起，用户点击“显示趋势图”展开；展开后页面滚动定位到图表区域

V1.5 样机建议用：

原生 Canvas（你已有）或 ECharts（更快更漂亮）

统一数据源：从 info_items + indicator_values 前端聚合（样机）；上线后改为后端聚合 API

5. API 设计（V1.5）
5.1 首页工作台

GET /api/domains/tree

GET /api/summary?root_domain_id=&current_domain_id=&from=&to=

返回：当前层级子域列表统计（去重口径）

GET /api/top-items?kind=info|alert&domain_id=&from=&to=&limit=10&sort=importance|time

5.2 情报总表

GET /api/info-items

filters:

domain_id（含子域）

from/to

is_new_tech

source_types[]

info_types[]

importance_min/max

has_active_alert

has_comment

q（全文检索）

sort & pagination

5.3 上传&复核（AI 两阶段）

POST /api/info-items/draft（保存草稿，字段齐全）

POST /api/info-items/{id}/ai-suggest

输入：title/content/source_url/source_types/info_types/domains/is_new_tech/importance/comment

输出：

indicators_suggestions[]

alert_suggestion: { should_alert, reasons[], color_suggested }

PUT /api/info-items/{id}（人工修订指标/字段）

POST /api/info-items/{id}/publish

服务端校验：

必填字段齐全

若 should_alert 或 manual alert：预警必填字段齐全

5.4 预警（人为新增/编辑）

POST /api/alerts（manual 创建）

PUT /api/alerts/{id}（编辑、延期）

POST /api/alerts/{id}/dismiss（解除+原因）

5.5 草稿管理

GET /api/info-items?status=draft&created_by=me

DELETE /api/info-items/{id}（草稿删除）

（样机阶段 drafts 可纯前端 localStorage，不必上 API）

5.6 Excel 批量上传

GET /api/templates/import.xlsx（下载模板）

POST /api/import/excel（解析预览，不直接发布）

输出：rows[] + errors[]

POST /api/import/excel/commit（批量生成 draft）

6. 前端实现方案（你先做样机也能无痛升级）

你现在是静态样机路线，我建议分两层：

6.1 样机阶段（纯前端 JSON）

所有新增功能仍然在前端实现，但把数据结构按“未来 DB/API”设计好（避免重写）

新增 ./data/info_item_domains.json（如果你不想改 info_items.json 结构）

或直接在 info_items.json 里加 domain_ids: [] + primary_domain_id

6.2 上线阶段（前后端分离）

前端只做展示与交互

汇总统计、全文检索、趋势聚合全部放后端

AI 推理由后端调用模型（并缓存）

7. AI 推荐指标 & 预警判定（可解释、可控）
7.1 AI 输出结构（建议）
{
  "indicators": [
    {
      "name": "捕集成本",
      "unit": "USD/tCO2",
      "suggested_value": 65,
      "confidence": 0.62,
      "evidence": "…原文片段…",
      "note": "来自摘要/新闻段落"
    }
  ],
  "alert": {
    "should_alert": true,
    "reasons": ["新技术", "成本显著低于历史范围下限"],
    "suggested_color": "orange",
    "explain": "…解释…"
  }
}

7.2 人工修订机制

每条指标 suggestion 可：接受/拒绝/改值/改单位

预警 suggestion 可：接受/关闭/改颜色

若预警最终为 true：强制填写预警正文 + reviewer_comment + duration

7.3 缓存（避免重复调用）

ai_indicator_suggestions 存模型输出

若草稿内容没变（hash_key 相同），直接复用上次 AI 结果

9. 权限与审计（你后面要给老板看的“可控”）

RBAC 不变（A/B/C），但新增两个关键控制点：

谁可以“人工开预警/解除预警”（B 可、A 不可、C 全局）

发布必须留下 reviewed_by（即使同一人复核，也要记录）

审计日志至少记录：

publish

alert create/update/dismiss

indicator range update（如果你后续启用范围）

10. 风险点与对策（工程实现层）

多领域导致统计重复

对策：统计按 info_id 去重；同时可展示关联数作为补充

筛选项很多，前端聚合会卡（样机数据一多就明显）

对策：上线后把筛选/聚合移到后端；样机用分页+惰性渲染

高亮与展开会带来 XSS 风险（如果 content 有 HTML）

对策：统一对 content 做 escape，再插入 <mark>（安全处理）

AI 建议与人工修改冲突

对策：AI 结果永远是“候选”，只有人工确认写入 indicator_values
